certificatechoice session

This method returns the signing certificate of the user and if needed queries the user which device they intend to use in the following signature session.

Having a correct certificate is needed for giving signatures under *AdES schemes. The method should not be used if the signature scheme does not mandate presence of the certificate in the document structure that is to be signed.

The user must confirm which device (and therefore signing certificate) they will use for the upcoming signing operation. How this is done depends on which endpoint/flow the RP has chosen:

  • in the dynamic link flow (the dynamic link based endpoint), the user interacts with the dynamic link (QR code, Web2App link, or App2App link) on a certain device, thereby indicating that this is the device that she intends to use.

  • in the notification based flow (the notification based endpoint), a certificate (device) choice dialogue is initiated on the user’s devices. The user interacts with one of the devices.

After the certificate has been determined, it becomes available to the RP via the session result endpoint.

In the notification based flow, if the user has only one account or the request references a :documentnumber, the certificate is returned to the RP without involving the mobile device at all.

The signature session that follows certificatechoice must always use the notification based endpoint. When the Smart-ID app receives a certificatechoice session it will service the request and then stay in the foreground and wait for the notification based signature session.

The method accepts QSCD as a certificate level parameter. This is a shortcut marking a certificate of QUALIFIED level which is also QSCD-capable. ADVANCED certificates cannot be QSCD-capable.

certificatechoice session endpoints

For the certificatechoice session requests, the API descriptions practically coincide. The only differences are:

  • Request endpoints

  • Response structure

Only the differing parts are described separately.

Dynamic link based certificatechoice session endpoints
Method URL

POST

BASE/v3/certificatechoice/dynamic-link/anonymous
Notification based certificatechoice session endpoints
Method URL

POST

BASE/v3/certificatechoice/notification/private/:issuer/:encoded-identifier

POST

BASE/v3/certificatechoice/notification/etsi/:id-etsi-qcs-SemanticsId-Natural

POST

BASE/v3/certificatechoice/notification/document/:documentnumber

Error conditions

  • HTTP error code 403 - Relying Party has no permission to issue the request. This may happen when:

    • Relying Party has no permission to invoke operations on accounts with ADVANCED certificates.

    • Relying Party has no permission to use requested capability.

  • HTTP error code 404 - object described in URL was not found, essentially meaning that the user does not have an account in Smart-ID system.

Request parameters

certificatechoice session request parameters
Parameter Type Mandatory Description

relyingPartyUUID

string

+

UUID of Relying Party.

relyingPartyName

string

+

RP friendly name, one of those configured for particular RP. Limited to 32 bytes in UTF-8 encoding.

certificateLevel

string

Level of certificate requested. ADVANCED/QUALIFIED/QSCD, defaults to QUALIFIED.

nonce

string

Random string, up to 30 characters. If present, must have at least 1 character.

capabilities

array

Used only when agreed with Smart-ID provider. When omitted request capabilities are derived from certificateLevel parameter.

requestProperties

object

A request properties object as a set of name/value pairs. IP sharing.
certificatechoice session request example
{
  "relyingPartyUUID": "1f1bfa89-4f8b-420a-a98e-fb3a161a30bc",
  "relyingPartyName": "DEMO",
  "certificateLevel": "QUALIFIED"
}

Response on successful session creation

The response depends on whether the notification based or dynamic link based endpoint was called.

Parameter Type Mandatory Description

sessionID

string

+

A string that can be used to request the operation result.

sessionToken

string

+

Unique random value that will be used to connect this certificate choice attempt between the relevant parties (RP, RP-API, mobile app)

sessionSecret

string

+

Base64-encoded random key value that should be kept in secret and should be shared only between the RP backend and RP-API server
